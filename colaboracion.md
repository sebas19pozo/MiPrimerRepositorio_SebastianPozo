 Parte 7: Investigación sobre colaboración en equipo con Git y GitHub

1.	¿Qué es un repositorio remoto y en qué se diferencia de uno local?
Un repositorio remoto es una versión del repositorio alojada en un servidor como GitHub, que permite la colaboración entre varios usuarios. Un repositorio local es la copia que se encuentra en el ordenador del desarrollador.

La diferencia es que el repositorio local se usa para trabajar individualmente y el remoto para compartir y sincronizar cambios.

2.	¿Qué es un "fork" en GitHub y cómo se usa en proyectos colaborativos?
Un fork es una copia de un repositorio en la cuenta de un usuario en GitHub. Se utiliza para trabajar sobre un proyecto sin modificar el original. Permite realizar cambios de forma independiente y luego proponerlos al repositorio original.

3.	¿Qué es un "pull request" (PR) y cuál es su propósito? Describe el flujo de trabajo típico.
Un pull request es una solicitud para incorporar cambios de una rama a otra dentro de un repositorio. Su propósito es permitir la revisión antes de integrar cambios.

El flujo típico es crear una rama, realizar cambios, subirla a GitHub, abrir el pull request, revisarlo y fusionarlo.

4.	¿Qué son los "issues" y los "proyectos" en GitHub? ¿Para qué sirven?
Los issues son herramientas para registrar errores, tareas o mejoras dentro de un repositorio. Sirven para organizar el trabajo y hacer seguimiento de tareas.

Los proyectos son tableros que permiten organizar issues y tareas en columnas para planificar el desarrollo.

5.	¿Cuál es la importancia de las ramas en un entorno de trabajo colaborativo y cómo se organizan en proyectos grandes (por ejemplo: main, dev, feature)?
Las ramas permiten trabajar en paralelo sin afectar la versión principal del proyecto. Esto evita errores en el código estable.
En proyectos grandes se suelen organizar ramas como main para la versión estable, dev para desarrollo, feature para nuevas funcionalidades y hotfix para correcciones urgentes.
